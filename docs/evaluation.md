# 검색 성능 평가

> 자동 계산: `docs/eval_queries.json`을 채운 뒤 `python docs/eval_runner.py` 실행하면 아래 표를 자동으로 채울 수 있음.
> 아래 수치는 실제로 `python docs/eval_runner.py`를 실행해서 얻은 결과이며(2026-07-15), 상품 데이터는 다나와에서 크롤링한 전자제품/가전 19,639건이다(1장 참고).

## 평가셋 구성

`docs/eval_queries.json`, 총 24개 쿼리, 유형별 8개씩:

| 유형 | 쿼리 예시 | 정답 상품 | 비고 |
|---|---|---|---|
| 정확 매칭 | MSI 벡터 A16 HX A8WHG-R9 | 상품 ID 1건 | 특정 SKU 하나만 정답 |
| 동의어/유사어 | 노트북 추천 | 상품 ID 5건 + `answer_category` | 카테고리 규모(411건) 대비 정답 ID는 일부만 |
| 오타/문맥 | 노트뷱, 전자렌지 | 상품 ID 2~5건 + `answer_category` | 오타/구어체 표현 |

동의어/오타 유형에는 `answer_category` 필드(DB에서 실제 조회한 정답 카테고리)를 추가로 채워서,
정답 ID를 못 맞혀도 "카테고리를 올바르게 이해했는가"를 별도로 측정할 수 있게 했다 (아래 3장 참고).

## 측정 지표

- **Recall@5 / Recall@10**: 상위 5개/10개 결과 안에 정답 상품 ID가 하나라도 포함된 쿼리 비율
- **Category Precision@5 / @10** *(추가)*: `answer_category`가 있는 쿼리에 한해, 상위 k개 결과 중 카테고리가 정답 카테고리와 일치하는 것의 비율

## 비교 결과

### 전체 Recall (기존 지표)

| 검색 방식 | Recall@5 | Recall@10 |
|---|---|---|
| BM25 단독 | 37.50% | 37.50% |
| BGE-M3 dense 벡터 단독 | 29.17% | 33.33% |
| 하이브리드 (BM25 + kNN) | 37.50% | 37.50% |

이 표만 보면 "벡터가 BM25보다 나쁘다"로 읽히는데, 아래 유형별 breakdown을 보면 이유가 다르다.

### 유형별 Recall@10 breakdown

| 유형 | BM25 | 벡터 | 하이브리드 |
|---|---|---|---|
| 동의어/유사어 | 0.00% | 0.00% | 0.00% |
| 오타/문맥 | 12.50% | 12.50% | 12.50% |
| 정확 매칭 | 100.00% | 87.50% | 100.00% |

- **동의어/오타 유형은 세 방식 모두 동일하게 낮다** → 방식 차이가 아니라 지표(정답 ID가 카테고리 규모 대비 너무 적음) 문제. 2장 케이스 스터디 참고.
- **정확 매칭에서만 벡터가 유일하게 처짐**(87.5% vs BM25/하이브리드 100%) → 근접 SKU를 구분 못하는 벡터의 실제 약점. 하이브리드가 완전히 구제.

### Category Precision@k (동의어/오타 16개 쿼리만 집계)

| 검색 방식 | CatP@5 | CatP@10 |
|---|---|---|
| BM25 단독 | 65.00% | 66.25% |
| **BGE-M3 dense 벡터 단독** | **92.50%** | **89.38%** |
| 하이브리드 (BM25 + kNN) | 65.00% | 66.25% |

"결과가 정답 카테고리에 실제로 속하는가"로 측정하면 벡터가 BM25를 큰 폭으로 앞선다. 하이브리드는 `multiMatch`(BM25)가 스코어를 주도해서 BM25 단독과 거의 동일한 수치가 나왔다 — kNN 후보군과 BM25 후보군이 잘 안 겹치는 케이스에서 하이브리드가 사실상 BM25처럼 동작한다는 [[env_gotchas]] 관찰과 일치.

## 케이스 스터디

### 1. "노트북" 단독 쿼리 — 벡터가 BM25보다 명확히 우수 (동의어/문맥 강점)

`GET /api/search/bm25?q=노트북` 상위 8건은 **전부 "노트북 백팩"류 상품**(RIVACASE, DAMONCOM 등, 카테고리 `패션잡화 > 가방`)이었다. 상품명에 "노트북"이라는 글자가 포함된 액세서리가 nori 형태소 분석 매치로 높은 점수를 받은 반면, 실제 노트북 상품명(`MSI 벡터 A16...`, `HP 15-fd2130TU` 등)에는 "노트북"이라는 단어 자체가 안 들어있어 BM25는 이걸 찾지 못했다.

같은 쿼리를 `GET /api/search?q=노트북`(벡터)로 보내면 상위 8건이 **전부 실제 노트북**(카테고리 `디지털 완제품 > 노트북`)이었다. 임베딩이 "노트북"이라는 단어의 의미(전자기기 카테고리)를 이해하고 있어서, 상품명에 그 단어가 문자 그대로 없어도 올바른 카테고리로 연결한다 — 이 프로젝트의 핵심 가설이 실제로 재현된 사례.

### 2. "MSI 벡터 A16 HX A8WHG-R9" 정확 매칭 — 벡터의 한계 (근접 SKU 구분 실패)

정답은 `id=1`("MSI 벡터 A16 HX A8WHG-R9 QHD+")인데, 카탈로그에는 같은 모델의 RAM/OS 구성만 다른 변형이 5개 더 있다(`83`: 32GB램, `121`: WIN11 32GB램, `1221`: WIN11, `1284`: WIN11 64GB램, `1289`: 64GB램). 벡터 검색은 이 6개 전부를 cosine 0.90~0.93으로 사실상 동점 처리해서 순위가 흔들리고, 실제 상위 결과는 `1289`, `1284`, `121`, `1221`, `1121`(R7 변형) 순으로 **정답 id=1이 top5 밖으로 밀렸다**.

반면 BM25는 쿼리에 없는 토큰("64GB램", "WIN11" 등)이 이름에 붙은 상품을 필드 길이 정규화로 감점시켜서 정확히 일치하는 `id=1`을 1위로 반환했다. 하이브리드도 동일하게 `id=1`을 1위로 반환 — 이 케이스에서는 BM25 성분이 하이브리드 스코어를 주도해서 벡터 단독의 약점을 구제했다.

**해석**: dense 임베딩은 "같은 모델, 스펙만 다른" SKU들을 의미적으로 거의 동일하게 취급한다. 이건 BGE-M3의 결함이라기보다, 다나와 카탈로그가 사람이 검색할 때 신경 안 쓰는 세부 스펙(RAM 용량 등)까지 별도 상품으로 세분화해놓은 데이터 특성과 맞물린 결과다.

### 3. "노트북 추천" 동의어형 쿼리 — Recall@k 지표 자체의 한계

정답으로 지정한 ID는 5개(`1,6,14,79,113`)뿐인데, 실제 "노트북" 카테고리에는 411개 상품이 있다(`SELECT count(*) FROM products WHERE category LIKE '%노트북%'`). 벡터 검색 결과 상위 10건을 직접 확인하니 **전부 실제 노트북**(의미적으로 완벽한 결과)이었지만, 411개 중 그 5개를 우연히 안 맞히면 recall@10 = 0으로 기록된다. 실제로 유형별 breakdown에서 동의어/유사어 recall@10이 BM25/벡터/하이브리드 전부 0.00%로 동일하게 나온 이유가 바로 이거다 — 세 방식의 실력 차이가 아니라, "카테고리 전체가 정답인데 극소수 ID만 정답표에 올린" 평가셋 설계의 한계다.

이 문제를 보완하기 위해 이번 세션에서 `answer_category` + category precision@k 지표를 추가했고(3장 표), 그 결과 벡터가 92.5%로 BM25(65%)를 크게 앞서는 게 드러났다 — recall@k에 가려져 있던 실제 성능 차이다.

## 결론

- **전체 recall@k만 보면 벡터(29~33%)가 BM25(37.5%)보다 낮아 원래 가설(벡터가 동의어/문맥에서 더 낫다)과 반대로 보이지만, 이는 지표 설계의 부작용이다.** 유형별로 쪼개보면:
  - 동의어/오타 유형은 세 방식이 recall@k에서 동률(0%, 12.5%)로 나오는데, 이는 정답 카테고리 규모(수백 건) 대비 정답 ID를 너무 적게 지정한 평가셋의 한계이지 검색 방식의 차이가 아니다.
  - 정확 매칭 유형에서만 벡터가 유일하게 뒤처진다(87.5% vs 100%) — 이는 실제 벡터의 약점(근접 SKU 구분 불가)이며, 하이브리드 검색이 완전히 보완한다.
- **Category Precision@k로 재측정하면 벡터(89~93%)가 BM25(65~66%)를 뚜렷하게 앞선다.** "노트북" 케이스 스터디(2장 1번)처럼, 상품명에 검색어가 문자 그대로 없어도 올바른 카테고리를 찾아내는 건 BM25가 원천적으로 못 하는 일이며, 이게 원래 가설이 검증하고자 했던 것이다.
- **종합**: BGE-M3 dense 벡터 검색은 "동의어/문맥 이해"에서 BM25보다 명확히 우수하다(가설 지지). 다만 "거의 동일한 상품의 세부 스펙 구분"에는 약하고, 이 약점은 하이브리드(BM25+kNN) 검색으로 보완된다. 실무 권장 방식은 **하이브리드** — 카테고리/문맥 이해는 벡터가, 정확한 SKU 매칭은 BM25가 담당하는 조합.
