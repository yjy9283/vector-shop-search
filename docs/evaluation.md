# 검색 성능 평가

> 자동 계산: `docs/eval_queries.json`을 채운 뒤 `python docs/eval_runner.py` 실행하면 아래 표를 자동으로 채울 수 있음.
> 아래 수치는 실제로 `python docs/eval_runner.py`를 실행해서 얻은 결과이며(2026-07-15, 하이브리드 RRF 적용 후 재측정),
> 상품 데이터는 다나와에서 크롤링한 전자제품/가전 19,639건이다(1장 참고).

## 평가셋 구성

`docs/eval_queries.json`, 총 24개 쿼리, 유형별 8개씩:

| 유형 | 쿼리 예시 | 정답 상품 | 비고 |
|---|---|---|---|
| 정확 매칭 | MSI 벡터 A16 HX A8WHG-R9 | 상품 ID 1건 | 특정 SKU 하나만 정답 |
| 동의어/유사어 | 노트북 추천 | 상품 ID 5건 + `answer_category` | 카테고리 규모(411건) 대비 정답 ID는 일부만 |
| 오타/문맥 | 노트뷱, 전자렌지 | 상품 ID 2~5건 + `answer_category` | 오타/구어체 표현 |

동의어/오타 유형에는 `answer_category` 필드(DB에서 실제 조회한 정답 카테고리)를 추가로 채워서,
정답 ID를 못 맞혀도 "카테고리를 올바르게 이해했는가"를 별도로 측정할 수 있게 했다 (아래 3장 참고).

## 측정 지표

- **Recall@5 / Recall@10**: 상위 5개/10개 결과 안에 정답 상품 ID가 하나라도 포함된 쿼리 비율
- **Category Precision@5 / @10** *(추가)*: `answer_category`가 있는 쿼리에 한해, 상위 k개 결과 중 카테고리가 정답 카테고리와 일치하는 것의 비율

## 하이브리드 검색 구현 - 점수 합산 → RRF로 교체

처음엔 ES 하나의 요청에 `knn` 절과 BM25 `query` 절을 같이 넣어서 점수를 그냥 더하는 방식으로 하이브리드를
구현했다. 그런데 벡터 점수(코사인, 0~1)와 BM25 점수(Lucene 스코어, 수십 단위)는 스케일이 완전히 달라서,
직접 확인해보니 "노트북" 쿼리에서 하이브리드 점수가 BM25 점수와 소수점까지 완전히 동일하게 나왔다 —
BM25가 조금이라도 매치되면 벡터 점수(최대 1점 정도)는 반올림 오차 수준으로 묻혀서 사실상 BM25 순위 그대로
나오는 문제였다.

이걸 **Reciprocal Rank Fusion(RRF)** 으로 교체했다 — 원점수가 아니라 "몇 등이었는가"만 보고
`RRF(d) = Σ 1/(k + rank)` (k=60, 관례값)로 합산하는 방식이라 점수 스케일 문제 자체가 없다. 벡터/BM25
각각 상위 50건씩 뽑아서 순위를 합치고, 상위 topK를 최종 반환한다(`SearchService.fuseRrf`).

**효과**: 아래 3장 표의 하이브리드 수치가 RRF 적용 전엔 BM25와 거의 동일했는데, 적용 후엔 Recall@10이
BM25/벡터 단독보다도 높아지고(37.5%→41.67%) Category Precision도 BM25/벡터 사이(66%→76.9%)로 자리잡아
"진짜 블렌드"가 됐다. 다만 트레이드오프도 있다 — "MSI 벡터 A16 HX A8WHG-R9" 정확매칭 쿼리에서 정답 id=1이
예전엔 하이브리드 1위였는데 RRF 후엔 4위로 내려갔다(BM25 단독 1위, 벡터 단독 8위라서 평균 순위가 밀림).
다행히 top5 안에는 남아있어서 Recall@5/@10엔 영향이 없었다.

## 비교 결과

### 전체 Recall

| 검색 방식 | Recall@5 | Recall@10 |
|---|---|---|
| BM25 단독 | 37.50% | 37.50% |
| BGE-M3 dense 벡터 단독 | 29.17% | 33.33% |
| 하이브리드 (RRF) | **41.67%** | **41.67%** |

RRF 적용 후 하이브리드가 BM25/벡터 단독보다 모두 높다 — 벡터가 못 잡는 걸 BM25가, BM25가 못 잡는 걸 벡터가
서로 보완해서 나오는 시너지.

### 유형별 Recall@10 breakdown

| 유형 | BM25 | 벡터 | 하이브리드 |
|---|---|---|---|
| 동의어/유사어 | 0.00% | 0.00% | **12.50%** |
| 오타/문맥 | 12.50% | 12.50% | 12.50% |
| 정확 매칭 | 100.00% | 87.50% | 100.00% |

동의어/유사어 유형에서 하이브리드가 유일하게 0%를 벗어났다 — RRF가 벡터와 BM25 양쪽의 상위권을 모두
후보로 살려두기 때문에, 좁은 정답 ID 중 하나라도 어느 한쪽 상위권에 있었다면 하이브리드가 건져올 수 있다.
정확 매칭은 여전히 100%(위 4장에서 설명한 순위 변동에도 불구하고 top10 안에는 계속 남아있음).

### Category Precision@k (동의어/오타 16개 쿼리만 집계)

| 검색 방식 | CatP@5 | CatP@10 |
|---|---|---|
| BM25 단독 | 65.00% | 66.25% |
| **BGE-M3 dense 벡터 단독** | **92.50%** | **89.38%** |
| 하이브리드 (RRF) | 80.00% | 76.88% |

RRF 적용 전엔 하이브리드가 BM25와 거의 같았는데(65~66%), 적용 후엔 BM25보다 확실히 높고 벡터보다는 낮은
"중간" 값(77~80%)이 나온다 — 두 방식의 카테고리 이해도가 실제로 섞였다는 뜻.

## 케이스 스터디

### 1. "노트북" 단독 쿼리 — 벡터가 BM25보다 명확히 우수 (동의어/문맥 강점)

`GET /api/search/bm25?q=노트북` 상위 8건은 **전부 "노트북 백팩"류 상품**(RIVACASE, DAMONCOM 등, 카테고리 `패션잡화 > 가방`)이었다. 상품명에 "노트북"이라는 글자가 포함된 액세서리가 nori 형태소 분석 매치로 높은 점수를 받은 반면, 실제 노트북 상품명(`MSI 벡터 A16...`, `HP 15-fd2130TU` 등)에는 "노트북"이라는 단어 자체가 안 들어있어 BM25는 이걸 찾지 못했다.

같은 쿼리를 `GET /api/search?q=노트북`(벡터)로 보내면 상위 8건이 **전부 실제 노트북**(카테고리 `디지털 완제품 > 노트북`)이었다. 임베딩이 "노트북"이라는 단어의 의미(전자기기 카테고리)를 이해하고 있어서, 상품명에 그 단어가 문자 그대로 없어도 올바른 카테고리로 연결한다 — 이 프로젝트의 핵심 가설이 실제로 재현된 사례.

**RRF 하이브리드로 같은 쿼리를 돌리면** 실제 노트북(`삼성전자 갤럭시북4`, `HP 15-FD1630TU`)과 노트북 백팩(`RIVACASE`, `DAMONCOM`)이 순위에 섞여서 나온다 — 벡터의 "의미 이해"와 BM25의 "키워드 매치"가 둘 다 반영된 결과. RRF 적용 전(점수 단순합산)에는 이 쿼리에서 하이브리드가 BM25와 완전히 동일하게 나왔었다.

### 2. "MSI 벡터 A16 HX A8WHG-R9" 정확 매칭 — 벡터의 한계, 그리고 RRF의 트레이드오프

정답은 `id=1`("MSI 벡터 A16 HX A8WHG-R9 QHD+")인데, 카탈로그에는 같은 모델의 RAM/OS 구성만 다른 변형이 5개 더 있다(`83`: 32GB램, `121`: WIN11 32GB램, `1221`: WIN11, `1284`: WIN11 64GB램, `1289`: 64GB램). 벡터 검색은 이 6개 전부를 cosine 0.90~0.93으로 사실상 동점 처리해서 순위가 흔들리고, id=1은 벡터 단독 순위에서 8위까지 밀린다.

BM25는 쿼리에 없는 토큰("64GB램", "WIN11" 등)이 이름에 붙은 상품을 필드 길이 정규화로 감점시켜서 정확히 일치하는 `id=1`을 1위로 반환한다. **점수를 단순 합산하던 예전 하이브리드는 BM25 점수가 워낙 커서 사실상 BM25 순위 그대로 나왔고, 결과적으로 id=1이 1위였다.** RRF로 바꾼 뒤에는 벡터 순위(8위)와 BM25 순위(1위)를 동등하게 평균 내다보니 id=1이 4위로 밀렸다 — 벡터 쪽에서 순위가 비슷한 근접 SKU(`1289`, `1221`, `1121`)들이 상대적으로 앞으로 온 것. 다행히 top5 안에는 남아있어서 Recall@5/@10 수치엔 영향이 없었다.

**해석**: dense 임베딩은 "같은 모델, 스펙만 다른" SKU들을 의미적으로 거의 동일하게 취급한다. RRF는 두 검색을 스케일 문제 없이 공정하게 섞어주지만, "이 케이스는 BM25가 훨씬 확신이 높다"는 정보까지는 반영하지 못한다 — 두 리트리버를 동등하게 신뢰하는 게 RRF의 기본 가정이기 때문. 더 다듬으려면 BM25 쪽에 가중치를 더 주는 weighted RRF를 시도해볼 수 있다.

### 3. "노트북 추천" 동의어형 쿼리 — Recall@k 지표 자체의 한계

정답으로 지정한 ID는 5개(`1,6,14,79,113`)뿐인데, 실제 "노트북" 카테고리에는 411개 상품이 있다(`SELECT count(*) FROM products WHERE category LIKE '%노트북%'`). 벡터 검색 결과 상위 10건을 직접 확인하니 **전부 실제 노트북**(의미적으로 완벽한 결과)이었지만, 411개 중 그 5개를 우연히 안 맞히면 recall@10 = 0으로 기록된다.

이 문제를 보완하기 위해 `answer_category` + category precision@k 지표를 추가했고(3장 표), 그 결과 벡터가 89.4%로 BM25(66.3%)를 크게 앞서는 게 드러났다 — recall@k에 가려져 있던 실제 성능 차이다.

## 결론

- **RRF로 바꾼 하이브리드가 BM25/벡터 단독보다 Recall@10(41.67%)·Category Precision@10(76.88%) 모두 더 낫다.** 벡터의 "의미 이해"와 BM25의 "정확 매치"가 실제로 서로를 보완하는 시너지를 확인했다.
- **Category Precision@k로 보면 벡터(89~93%)가 BM25(65~66%)를 뚜렷하게 앞선다.** "노트북" 케이스 스터디처럼, 상품명에 검색어가 문자 그대로 없어도 올바른 카테고리를 찾아내는 건 BM25가 원천적으로 못 하는 일이며, 이게 원래 가설이 검증하고자 했던 것이다.
- **다만 RRF는 만능이 아니다.** "거의 동일한 상품의 세부 스펙 구분"이 필요한 정확매칭 쿼리에서는, 두 리트리버를 동등하게 신뢰하는 RRF의 기본 가정 때문에 확신도가 높은 BM25의 1위 판정이 흔들릴 수 있다(케이스 스터디 2번). 다행히 이번 평가셋에서는 top5 밖으로 밀리는 경우는 없었다.
- **종합**: BGE-M3 dense 벡터 검색은 "동의어/문맥 이해"에서 BM25보다 명확히 우수하다(가설 지지). RRF 기반 하이브리드는 이 강점과 BM25의 정확 매칭 강점을 실제로 함께 반영하는 균형 잡힌 조합이며, 세 방식 중 실무에 가장 적합하다.
